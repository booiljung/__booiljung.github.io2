생명주기 노드 기본 구조

ROS2 생명주기 노드는 rclcpp_lifecycle 패키지를 통해 구현할 수 있다. 기본적으로 생명주기 노드는 상태(state)에 따라 동작하며, 주요 상태는 "unconfigured", "inactive", "active", "finalized"이다. 각 상태에서 특정 트랜지션(transition)을 통해 상태가 변환되며, 트랜지션에 따라 콜백 함수가 호출된다.

#include "rclcpp/rclcpp.hpp"
#include "rclcpp_lifecycle/lifecycle_node.hpp"

class LifecycleExample : public rclcpp_lifecycle::LifecycleNode
{
public:
    LifecycleExample()
    : rclcpp_lifecycle::LifecycleNode("lifecycle_example")
    {
        RCLCPP_INFO(this->get_logger(), "Lifecycle node constructor called");
    }

    using CallbackReturn = rclcpp_lifecycle::node_interfaces::LifecycleNodeInterface::CallbackReturn;

    CallbackReturn on_configure(const rclcpp_lifecycle::State & state)
    {
        RCLCPP_INFO(this->get_logger(), "on_configure() is called.");
        return CallbackReturn::SUCCESS;
    }

    CallbackReturn on_activate(const rclcpp_lifecycle::State & state)
    {
        RCLCPP_INFO(this->get_logger(), "on_activate() is called.");
        return CallbackReturn::SUCCESS;
    }

    CallbackReturn on_deactivate(const rclcpp_lifecycle::State & state)
    {
        RCLCPP_INFO(this->get_logger(), "on_deactivate() is called.");
        return CallbackReturn::SUCCESS;
    }

    CallbackReturn on_cleanup(const rclcpp_lifecycle::State & state)
    {
        RCLCPP_INFO(this->get_logger(), "on_cleanup() is called.");
        return CallbackReturn::SUCCESS;
    }

    CallbackReturn on_shutdown(const rclcpp_lifecycle::State & state)
    {
        RCLCPP_INFO(this->get_logger(), "on_shutdown() is called.");
        return CallbackReturn::SUCCESS;
    }
};

int main(int argc, char * argv[])
{
    rclcpp::init(argc, argv);

    auto lc_node = std::make_shared<LifecycleExample>();

    rclcpp::spin_some(lc_node->get_node_base_interface());
    rclcpp::shutdown();
    return 0;
}

생명주기 상태 전환

생명주기 노드는 아래와 같은 상태 전환을 지원한다.

  • Unconfigured 상태: 노드가 초기화되지만, 아직 활성화되지 않은 상태이다. on_configure() 콜백 함수가 호출되며, 노드의 초기 설정을 담당한다.

  • Inactive 상태: 노드가 구성되었으나 아직 데이터를 처리하거나 퍼블리셔를 활성화하지 않은 상태이다. on_activate() 콜백을 통해 이 상태로 전환될 수 있다.

  • Active 상태: 노드가 완전히 활성화된 상태이다. 이때 노드는 데이터를 퍼블리시하거나 수신하는 등의 작업을 할 수 있다.

  • Deactivated 상태: 노드가 비활성화된 상태이다. 이 상태에서는 노드가 더 이상 데이터를 처리하지 않으며, on_deactivate() 콜백을 통해 이 상태로 전환된다.

  • Finalized 상태: 노드가 완전히 종료된 상태이다. on_cleanup()이나 on_shutdown() 콜백을 통해 이 상태로 전환된다.

상태 전환 수식

생명주기 상태 전환을 수식으로 표현하면, 각 상태는 노드의 상태 \mathbf{S}(t)로 표현될 수 있다. 여기서 t는 시간이고, 각 상태는 \mathbf{S}_i로 정의된다.

\mathbf{S}(t) = \begin{cases} \mathbf{S}_1, & \text{if } t \in [t_0, t_1) \\ \mathbf{S}_2, & \text{if } t \in [t_1, t_2) \\ \mathbf{S}_3, & \text{if } t \in [t_2, t_3) \\ \mathbf{S}_4, & \text{if } t \in [t_3, t_4) \\ \mathbf{S}_5, & \text{if } t \in [t_4, t_5) \\ \end{cases}

여기서 각 상태 \mathbf{S}_i는 다음과 같이 정의된다:

\mathbf{S}_1 = \text{Unconfigured 상태}
\mathbf{S}_2 = \text{Inactive 상태}
\mathbf{S}_3 = \text{Active 상태}
\mathbf{S}_4 = \text{Deactivated 상태}
\mathbf{S}_5 = \text{Finalized 상태}

이 상태들은 트랜지션 함수 T_{ij}에 의해 전환되며, 각 트랜지션은 시간에 따라 변한다. 예를 들어, \mathbf{S}_1에서 \mathbf{S}_2로의 트랜지션은 T_{12}(t)로 표현된다.

T_{12}(t) = \text{on\_configure}(\mathbf{S}_1, t)
T_{23}(t) = \text{on\_activate}(\mathbf{S}_2, t)

이러한 상태 전환을 통해 노드는 서로 다른 상태에서 다양한 작업을 수행할 수 있다.

상태 전환 다이어그램

생명주기 노드의 상태 전환은 상태 다이어그램으로 표현될 수 있으며, 각 상태와 트랜지션을 다이어그램으로 명확히 시각화할 수 있다.

다음은 기본적인 상태 전환 다이어그램이다:

+----------------+    on_configure()     +----------------+
| Unconfigured   |---------------------->| Inactive       |
+----------------+                       +----------------+
                         on_activate()
                               |
                               v
                        +----------------+
                        | Active         |
                        +----------------+
                               |
                        on_deactivate()
                               |
                               v
                        +----------------+
                        | Inactive       |
                        +----------------+
                               |
                        on_cleanup() or
                        on_shutdown()
                               |
                               v
                        +----------------+
                        | Finalized      |
                        +----------------+

노드 활성화 및 비활성화

on_activate()와 on_deactivate()는 생명주기 노드에서 가장 중요한 콜백 중 하나이다. 각각 활성화 및 비활성화 상태로의 전환을 담당하며, 노드의 퍼블리셔, 서브스크라이버 등의 통신을 관리한다.

활성화 예제

노드를 활성화하면 퍼블리셔가 활성화되어 데이터를 퍼블리시할 수 있게 된다. 아래는 노드 활성화 시 퍼블리셔를 시작하는 예제이다:

CallbackReturn on_activate(const rclcpp_lifecycle::State & state) override
{
    RCLCPP_INFO(this->get_logger(), "on_activate() called.");
    publisher_->on_activate();  // 퍼블리셔 활성화
    return CallbackReturn::SUCCESS;
}

비활성화 예제

노드를 비활성화하면 퍼블리셔가 데이터를 더 이상 전송하지 않게 된다. 이를 통해 시스템 자원을 절약하고, 불필요한 통신을 막을 수 있다.

CallbackReturn on_deactivate(const rclcpp_lifecycle::State & state) override
{
    RCLCPP_INFO(this->get_logger(), "on_deactivate() called.");
    publisher_->on_deactivate();  // 퍼블리셔 비활성화
    return CallbackReturn::SUCCESS;
}

이와 같이, 생명주기 노드는 노드가 상태에 따라 퍼블리셔나 서브스크라이버의 활성화 상태를 조정할 수 있다.

퍼블리셔와 서브스크라이버 관리

ROS2 생명주기 노드에서는 퍼블리셔와 서브스크라이버를 활성화하거나 비활성화할 수 있다. 이 과정에서 중요한 것은 퍼블리셔와 서브스크라이버의 on_activate()와 on_deactivate() 메서드를 적절히 호출하여 상태를 관리하는 것이다.

퍼블리셔의 활성화와 비활성화

퍼블리셔는 노드가 Active 상태에 있을 때만 데이터를 퍼블리시할 수 있다. on_activate() 메서드를 통해 퍼블리셔를 활성화하고, on_deactivate() 메서드를 통해 비활성화한다.

class LifecyclePublisherExample : public rclcpp_lifecycle::LifecycleNode
{
public:
    LifecyclePublisherExample()
    : rclcpp_lifecycle::LifecycleNode("lifecycle_publisher_example")
    {
        // 퍼블리셔 생성
        publisher_ = this->create_publisher<std_msgs::msg::String>("topic", 10);
    }

    using CallbackReturn = rclcpp_lifecycle::node_interfaces::LifecycleNodeInterface::CallbackReturn;

    CallbackReturn on_activate(const rclcpp_lifecycle::State & state)
    {
        RCLCPP_INFO(this->get_logger(), "Publisher activated.");
        publisher_->on_activate();  // 퍼블리셔 활성화
        return CallbackReturn::SUCCESS;
    }

    CallbackReturn on_deactivate(const rclcpp_lifecycle::State & state)
    {
        RCLCPP_INFO(this->get_logger(), "Publisher deactivated.");
        publisher_->on_deactivate();  // 퍼블리셔 비활성화
        return CallbackReturn::SUCCESS;
    }

private:
    rclcpp_lifecycle::LifecyclePublisher<std_msgs::msg::String>::SharedPtr publisher_;
};

이 예제에서 LifecyclePublisherExample 클래스는 on_activate() 메서드를 통해 퍼블리셔를 활성화하고, on_deactivate() 메서드를 통해 퍼블리셔를 비활성화한다.

상태 전환을 위한 서비스 호출

생명주기 노드에서 상태를 전환하는 것은 외부에서 트리거할 수도 있다. 이를 위해 ROS2는 생명주기 노드가 상태 전환 서비스를 제공하도록 지원한다. 이러한 상태 전환 서비스는 노드 외부에서 상태를 변경하고자 할 때 사용된다.

상태 전환 서비스 예제

rclcpp::Client<rclcpp_lifecycle::ChangeState>::SharedPtr change_state_client_;
change_state_client_ = this->create_client<rclcpp_lifecycle::ChangeState>(
    "/lifecycle_example/change_state");

auto request = std::make_shared<rclcpp_lifecycle::ChangeState::Request>();
request->transition.id = rclcpp_lifecycle::Transition::TRANSITION_ACTIVATE;

if (change_state_client_->wait_for_service(std::chrono::seconds(1))) {
    auto future_result = change_state_client_->async_send_request(request);
    if (rclcpp::spin_until_future_complete(this->get_node_base_interface(), future_result) ==
        rclcpp::FutureReturnCode::SUCCESS)
    {
        RCLCPP_INFO(this->get_logger(), "State changed to active.");
    }
} else {
    RCLCPP_ERROR(this->get_logger(), "Service not available.");
}

위 코드에서, change_state_client_는 노드의 상태 전환 서비스를 호출한다. 이 예제에서는 노드를 "active" 상태로 전환하는 트랜지션을 요청하고, 서비스가 성공적으로 호출되면 상태가 "active"로 변경된다.

상태 콜백 함수의 활용

생명주기 노드의 상태 콜백 함수는 상태 전환 시 발생하는 동작을 정의하는 곳이다. 각 콜백 함수는 노드가 각 상태에서 어떤 작업을 할지를 결정한다.

상태 콜백 함수 예제

CallbackReturn on_cleanup(const rclcpp_lifecycle::State & state) override
{
    RCLCPP_INFO(this->get_logger(), "Cleaning up resources.");
    // 필요한 리소스 정리 작업
    return CallbackReturn::SUCCESS;
}

CallbackReturn on_shutdown(const rclcpp_lifecycle::State & state) override
{
    RCLCPP_INFO(this->get_logger(), "Shutting down node.");
    // 노드 종료 시 필요한 작업 처리
    return CallbackReturn::SUCCESS;
}

위의 on_cleanup()과 on_shutdown() 콜백 함수는 각각 노드의 자원을 정리하거나 종료할 때 호출된다. 이를 통해 노드는 안전하게 종료되며, 자원을 적절히 해제할 수 있다.

상태 전환의 시간 복잡도

상태 전환이 발생할 때마다 노드에서 실행되는 콜백 함수는 다음과 같은 시간 복잡도를 갖는다:

  1. on_configure(): 노드의 초기 설정을 처리하며, 이는 대부분 상수 시간 복잡도 \mathcal{O}(1)를 갖는다.
  2. on_activate(): 퍼블리셔나 서브스크라이버를 활성화하는 작업을 처리하는데, 이 역시 \mathcal{O}(1)의 시간 복잡도를 가질 수 있다.
  3. on_deactivate(): 퍼블리셔나 서브스크라이버를 비활성화하며, \mathcal{O}(1)로 처리된다.
  4. on_cleanup(): 자원을 해제하는 과정으로, 자원의 수에 따라 시간 복잡도는 \mathcal{O}(n)이 될 수 있다. 여기서 n은 자원의 수를 나타낸다.
  5. on_shutdown(): 노드를 완전히 종료하는 작업이므로 자원 정리와 유사하게 \mathcal{O}(n)의 시간 복잡도를 가질 수 있다.

이처럼 상태 전환 시 각 콜백 함수는 노드의 상태와 자원에 따라 적절히 시간 복잡도가 결정된다.

생명주기 노드의 실시간 적용

실시간 시스템에서 생명주기 노드는 매우 중요한 역할을 한다. 노드가 "active" 상태에서만 실제 데이터를 처리하게 하여, 실시간 시스템의 자원 소모를 최소화하고 예측 가능한 실행 흐름을 유지할 수 있다.

실시간 시스템에서의 활용 예제

실시간 시스템에서의 생명주기 노드는 노드가 "inactive" 상태에서는 자원을 최소한으로 소모하고, "active" 상태에서만 작업을 처리한다. 이를 통해 시스템의 부하를 조절할 수 있다.

CallbackReturn on_activate(const rclcpp_lifecycle::State & state) override
{
    RCLCPP_INFO(this->get_logger(), "Activating node for real-time operations.");
    // 실시간 작업 시작
    start_real_time_processing();
    return CallbackReturn::SUCCESS;
}

CallbackReturn on_deactivate(const rclcpp_lifecycle::State & state) override
{
    RCLCPP_INFO(this->get_logger(), "Deactivating node to conserve resources.");
    // 실시간 작업 중단
    stop_real_time_processing();
    return CallbackReturn::SUCCESS;
}

위 예제에서 on_activate()는 실시간 작업을 시작하며, on_deactivate()는 실시간 작업을 중단하여 시스템 자원을 절약하는 방법을 보여준다.
